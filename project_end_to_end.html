Technical summary: what this project proves

Final model

XGBoost

Primary applicant-facing model for tabular credit-card default risk.

Primary dataset

30,000 rows

UCI Default of Credit Card Clients, Taiwan credit-card default.

Operating policy

0.25 threshold

Validation-selected threshold for default-risk manual-review screening.

The project uses the public UCI Taiwan credit-card default dataset to build a responsible credit-risk workflow. The model estimates next-month default risk, explains the drivers of that risk, audits leakage and fairness concerns, compares machine-learning and deep-learning alternatives, and exposes the final workflow in a Streamlit dashboard.

The final conclusion is deliberately practical: XGBoost remains the primary model because it performs well on structured tabular data, beats the DNN benchmark on ranking quality, and is easier to explain with SHAP and LIME. The recall-optimized policy keeps the same XGBoost model but lowers the classification threshold from 0.50 to 0.25 for manual-review screening, increasing test-set default recall from 34.14% to 58.10%.

Important boundary: this is an educational and portfolio-grade decision-support project. It is not a production credit approval engine, a regulatory scorecard, an adverse-action reason generator, or a substitute for legal, compliance, model-risk, or underwriting review.

Business problem and why the design makes sense

Credit-risk teams care about identifying customers who may default while avoiding unnecessary friction for customers who are likely to repay. A model that only maximizes accuracy can be misleading because defaults are the business-critical minority class. Missing actual defaulters can be costly, but flagging too many reliable customers creates manual-review workload and customer-experience friction.

Business objective

Support a human credit-risk review workflow by estimating default risk, highlighting the strongest drivers, and showing how threshold choice changes default capture, precision, and review volume.

Why recall matters

Recall measures how many actual defaulters are captured. The recall policy captures more risky cases for review, accepting more false positives as an explicit operational tradeoff rather than hiding that tradeoff behind accuracy.

Why explanations matter

A risk score is not enough for review. SHAP, LIME, and counterfactual-style guidance help translate the score into repayment, utilization, bill, and payment drivers that a reviewer can reason about.

Why governance matters

Leakage checks, fairness metrics, ML-vs-DL comparisons, and threshold reports make the result defensible. They also make clear what the project does not prove.

Dataset: public UCI Taiwan credit-card default data

The primary dataset is the UCI Default of Credit Card Clients dataset, loaded through ucimlrepo with UCI dataset ID 350. The target is standardized in the project as Default_Flag, where 1 means the customer defaulted in the next month.

Preprocessing and leakage controls

The pipeline standardizes column names, coerces numeric UCI fields, creates engineered credit-risk features, removes ID-style columns, normalizes the binary target, and builds a scikit-learn preprocessing pipeline with median imputation and scaling for numeric features plus most-frequent imputation and one-hot encoding for categorical features.

  1. Load data. src.data_api_loader supports UCI, local, and URL sources. The final project path uses the UCI source by default.
  2. Normalize schema. UCI columns such as X1 and Y are mapped to project names such as LIMIT_BAL and Default_Flag.
  3. Engineer features. The pipeline creates utilization, payment-ratio, delay-summary, average bill, and average payment features.
  4. Select policy-compliant features. The application_public feature set excludes SEX from active training but keeps it available for fairness analysis.
  5. Split and train. The final saved metrics use a held-out split with train/test separation and no threshold selection on the test set.
Leakage audit result: no detected target leakage or train/test overlap based on implemented checks. The target-shuffle ROC-AUC is 0.4922, close to random, which supports the conclusion that the pipeline is not learning a hidden copy of the target.
Leakage check Observed result Interpretation
Target excluded from model features True The supervised label is not passed as an input.
ID columns excluded from model features True ID-style identifiers are not used as predictors.
Source-index train/test overlap 0 No direct row-index overlap was detected.
Duplicate selected rows across train/test 0 No duplicate selected-row leakage was detected.

Models compared and why each exists

The project compares a transparent linear benchmark, a tree-boosting primary model, and a neural-network benchmark. This structure is intentional: it shows that the final model was selected against alternatives rather than assumed.

Model Role Why included Key implementation detail
Logistic regression Transparent benchmark Simple, explainable baseline for comparison. class_weight="balanced", max_iter=1500.
XGBoost Primary model Strong on structured tabular data and compatible with SHAP/LIME explanations. 150 trees, max depth 4, learning rate 0.05, subsample 0.9, column sample 0.9.
Recall-optimized XGBoost policy Manual-review screening policy Uses the same model but changes the operating threshold to capture more defaults. Threshold 0.25 selected on validation data only.
TensorFlow/Keras DNN Governance benchmark Tests whether extra complexity improves ranking, recall, or PR-AUC enough to justify opacity. MLP architecture 64 → 32 → 16 → 1.
Final model decision: XGBoost remains primary because it has stronger ROC-AUC and PR-AUC than the DNN baseline, better recall-policy default capture than the DNN recall policy, and better operational explainability.

Metrics and final results

Accuracy is reported, but it is not the main decision metric. Default risk is an imbalanced classification problem, so the project focuses on precision, recall, F1, F2, ROC-AUC, PR-AUC, and the non-default support rate implied by threshold choice.

Metric Meaning Why it matters here
Precision Of customers flagged as default risk, how many actually defaulted? Measures review efficiency and false-positive burden.
Recall Of actual defaulters, how many were captured? Directly tied to default-risk screening value.
F2 F-score that weights recall more than precision. Useful when missing actual defaults is more costly than sending more cases to review.
ROC-AUC Overall ranking quality across thresholds. Useful model-comparison measure, but can look optimistic in imbalanced settings.
PR-AUC Precision-recall ranking quality for the positive/default class. More directly tied to minority-class detection quality.

What improved with the recall policy?

Default-class recall improved from 34.14% at the XGBoost 0.50 baseline threshold to 58.10% at the 0.25 recall-policy threshold. That means more actual defaulters are routed to manual review.

What tradeoff did it create?

Precision dropped from 65.84% to 47.77%, and the non-default support rate dropped from 88.54% to 73.11%. This is the expected cost of a more sensitive screening policy.

Threshold tuning: how the operating policy was selected

The model produces probabilities. A threshold turns those probabilities into a screening flag. The default 0.50 threshold is not a business law; it is just a conventional cutoff. This project tunes the threshold because the business objective is to capture more actual defaults for manual review.

  1. Create an untouched test split. Test data is held out from selection.
  2. Use inner-train and validation data. Candidate thresholds are evaluated on validation data, not on test data.
  3. Apply the primary selection rule. Select the threshold that maximizes recall while maintaining validation precision of at least 0.50.
  4. Evaluate once on test. The selected threshold is then evaluated on the held-out test split to estimate out-of-sample performance.

Explainability: concepts used and why they are useful

Explainability is not decorative in this project. It is part of the business workflow: a reviewer needs to understand why a borrower is flagged, which drivers are important, and what kind of behavior change might lower model-estimated risk.

Technique What it explains Why used here Caveat
SHAP Global and local contribution of each feature to XGBoost predictions. Strong fit for tree models; supports applicant-level reason summaries. Explains the model, not causal reality.
LIME Local approximation around one applicant prediction. Provides a second local explanation view to compare with SHAP. Local surrogate explanations can vary by sampling choices.
Counterfactual-style guidance Scenario-based changes in repayment, utilization, and exposure assumptions. Converts model behavior into practical improvement guidance. It is decision-support guidance, not a guaranteed outcome.
DNN permutation importance Drop in PR-AUC when one feature is permuted. Model-agnostic fallback for a more opaque neural network benchmark. Approximate and diagnostic; not a replacement for tree-model explanations.

Top model drivers are economically plausible

The leading XGBoost importance and DNN permutation-importance signals emphasize repayment delay, number of delayed months, bill-to-limit behavior, payments, and credit exposure. This is economically plausible: customers with recent and repeated delays, higher utilization, or weak repayment amounts should carry higher near-term default risk.

Evidence source Leading drivers Interpretation
XGBoost feature importance RecentPaymentDelay, MaxPaymentDelay, PAY_0, NumDelayedMonths, AvgBillAmount Tree model focuses on recent delinquency and balance behavior.
DNN permutation importance PAY_0, RecentPaymentDelay, NumDelayedMonths, PAY_6, BillToLimitRatio_2 Neural benchmark points to similar repayment and utilization signals.

Fairness and governance: what was audited

The protected attribute used for fairness diagnostics is SEX/gender. The verified UCI coding is SEX=1 for Male and SEX=2 for Female. The favorable outcome is predicted non-default / lower-risk manual-review support. The active XGBoost training feature set excludes SEX, but the audit data retains it so group-level diagnostics can be computed.

Did we find discrimination? We did not prove legal discrimination or causal bias. What we did find is a diagnostic fairness-governance signal. Male applicants (SEX=1) had higher high-risk flag rates than Female applicants (SEX=2), and Female applicants had higher false-negative rates. The protected attribute was not used directly in the active prediction path, but changing the threshold still widened group-level fairness metrics enough that a real lender would need governance review before using that policy.

XGBoost baseline 0.50 0.0220 0.0063 0.0225 0.9754
XGBoost recall policy 0.25 0.0691 0.0468 0.0723 0.9089

Deeper protected-attribute fairness analysis

The deep-dive module adds group outcome analysis, error-rate audit, calibration check, proxy-risk analysis, feature association diagnostics, SHAP driver comparison, threshold fairness frontier, individual SEX sensitivity, and a nearest-neighbour individual fairness diagnostic. These reports preserve both numeric SEX codes and readable Male/Female labels. They are governance diagnostics; they are not proof of legal discrimination and make no causal bias claim.

Outcome differences by verified group

At threshold 0.50, high-risk flag rates were 0.1280 for Male applicants (SEX=1) and 0.1060 for Female applicants (SEX=2). At threshold 0.25, they increased to 0.3109 and 0.2419. Male applicants had higher high-risk flag rates at both thresholds. This is a threshold governance signal, not a legal conclusion.

Error harms differ by group

At threshold 0.50, Male FPR was 0.0542 versus Female FPR 0.0479, while Female FNR was 0.6684 versus Male FNR 0.6459. At threshold 0.25, Male FPR was 0.2094 versus Female FPR 0.1626, while Female FNR was 0.4505 versus Male FNR 0.3782. Higher FPR affects actual non-defaulters through more high-risk flags; higher FNR increases lender exposure by missing actual defaulters.

Threshold fairness frontier

Lowering XGBoost from 0.50 to 0.25 improved recall from 0.3414 to 0.5810, but widened demographic parity difference from 0.0220 to 0.0691 and equalized odds difference from 0.0225 to 0.0723. Threshold choice is therefore a governance decision, not only a model-performance decision.

Proxy-risk analysis

A random-forest proxy model predicted SEX/gender from non-sensitive variables with ROC-AUC 0.6476. Top proxy-associated variables included BillToLimitRatio_1, BillToLimitRatio_2, PAY_2, AvgBillToLimitRatio, and BillToLimitRatio_3. This means removing SEX is necessary for direct-use control but not sufficient as a complete fairness strategy.

Direct-use sensitivity

Flipping SEX only produced maximum probability change 0.00000000 and zero baseline or recall-policy decision changes. This verifies no direct use of SEX in the active XGBoost prediction path, but it does not rule out proxy effects through other variables.

Calibration and individual fairness

The calibration check found a largest bin-level calibration gap of 0.0863. The nearest-neighbour diagnostic found median cross-group score difference 0.0278 and 90th percentile difference 0.1156. Both are diagnostic fairness-risk signals, not conclusive legal or causal findings.

Interpretation: the protected-attribute deep dive does not establish legal discrimination or causal bias. It shows a diagnostic fairness-governance signal: Male applicants (SEX=1) had higher high-risk flag rates, Female applicants (SEX=2) had higher false-negative rates, the recall-focused threshold improved default capture while widening several fairness metrics, and proxy signal remains possible even though SEX is not directly used in the active XGBoost prediction path. The model should remain a decision-support tool requiring threshold governance, calibration monitoring, proxy-risk review, and human oversight before any real operational use.

Complete roadmap from the user's perspective

The project has two user journeys: a repo reviewer journey and a dashboard user journey. The reviewer journey explains how an outsider should inspect or reproduce the work. The dashboard journey explains how a non-technical reviewer or credit-risk stakeholder uses the local app.

Repository reviewer roadmap

  1. Start here. Read this HTML file for the complete narrative, then read README.md for quick setup commands.
  2. Check governance summaries. Review docs/model_card.md and docs/interview_defense.md.
  3. Inspect saved results. Open reports/final_project_report.md, ML-vs-DL comparison, recall summary, leakage audit, and fairness comparison.
  4. Review source implementation. Key files are src/data_api_loader.py, src/dataset_adapters.py, src/data_preprocessing.py, src/train_xgboost.py, src/recall_optimization.py, src/shap_explainer.py, src/lime_explainer.py, and dashboard/app.py.
  5. Reproduce if needed. Install dependencies, run the pipeline, run tests, and launch the dashboard using the commands below.

Dashboard user roadmap

  1. Open Applicant Report. Enter a compact applicant profile with target credit amount, repayment status, average bill amount, and average payment amount.
  2. Generate the risk report. The dashboard returns predicted default risk, risk band, manual-review signal, and maximum advisable credit exposure.
  3. Read the explanation. The report translates model drivers into repayment, utilization, and payment behavior language.
  4. Use Improvement Guidance. Review shortcomings, why each issue matters, and simulate changes such as lower exposure, better repayment delay, higher average payment, or lower bill amount.
  5. Use Model Governance. Technical users inspect XGBoost metrics, threshold tradeoffs, DNN benchmark results, fairness diagnostics, leakage audit, and explainability artifacts.

Limitations and future scope

The project is intentionally honest about what it has not solved. That is important for credit-risk work because a clean prototype is not the same as a production lending system.

Current limitations

  • Public academic dataset, not production bank data.
  • No true chronological application timestamp for time-based validation.
  • No reject inference or application population correction.
  • No calibrated regulatory scorecard.
  • No production monitoring, access control, drift governance, or legal workflow.
  • Fairness metrics are diagnostic and observational, not causal or compliance proof.

Future roadmap

  • Add probability calibration and calibration monitoring.
  • Add threshold governance with review-capacity and cost-sensitivity inputs.
  • Build a parallel scorecard track with WOE/IV/binning/PDO/base odds.
  • Add true time-based validation if timestamped application data becomes available.
  • Add intersectional fairness diagnostics where sample sizes support them.
  • Add drift monitoring, challenger-model review, and model-risk governance controls.
  • Validate on additional public credit-risk datasets as separate external checks.
